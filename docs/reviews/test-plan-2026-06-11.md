## Edge Cases
- SECURITY ★ undefined session claims → request denied (fail-closed), never unfiltered query
- SECURITY ★ cross-account patient id → 404 (no existence leak); deactivated user → 403 immediately
- SECURITY ★ X-Account-Id not in claims → 404/403; claims re-minted on refresh after account reassignment
- Parser: BOM-prefixed CSV; Windows-1252 encoding; XLSX serial dates (epoch 1899-12-30, no TZ shift on DOB); multi-sheet workbook (rule: first non-empty sheet, else validation error); duplicate headers; blank rows; formula cells; 10MB/row-cap/zip-bomb rejection; empty file; zero valid rows
- Validation: unknown internal_status; owner/phase canonicalization ("Provider Office " → "Provider office"); duplicate referral_id in file; bad dates per-row
- Matching: NPI > alias > exact-name priority; normalized alias collision rejected; "Smith-Jones" hyphen case; did-you-mean suggestions present
- Publish: transaction rollback mid-batch leaves zero changes; concurrent publish (row lock, VALIDATED→PUBLISHING→PUBLISHED); referral moved between accounts → old row deactivated + shown in diff; scoped deactivation default OFF
- Predicates: needsAction = owner=='Provider office'; newToday from latest published batch via typed StatusEvents (CREATED/STATUS_CHANGED/PHASE_CHANGED/OWNER_CHANGED)
- User provisioning: ST signup succeeds + Prisma fails → compensating cleanup, no orphan login
- DOB: @db.Date round-trip renders identical date in EST/PST (no off-by-one)

## Critical Paths
1. Cross-tenant isolation suite (the product's stated zero-incident metric) — integration, ship-blocking
2. Publish pipeline (upload→validate→preview→publish→extension shows data) — E2E with seeded backend
3. Fail-closed auth (undefined claims, deactivated user, wrong account header) — integration, ship-blocking
4. Parser fixture suite (real-world hospital export fixtures incl. BOM/encoding/serial dates) — unit
5. Provider daily flow (login→needs-action→detail→sign-out) — E2E (Playwright, extension context)

## Test pyramid & flakiness
- Many unit (parser, matcher, validation, predicates, normalizeName) → fewer integration (API + DB + ST docker core) → 3 E2E flows
- Flakiness watch: anything touching the ST core (use local docker, never try.supertokens.io); time-dependent stale-banner tests (inject clock); no test depends on row ordering
- LLM/eval suites: none (no LLM surface in V1)

## 2am-Friday confidence tests
- Hostile QA: upload a file where account_name = an alias of the WRONG account with a colliding normalization → publish must be blocked pending admin confirm
- Chaos: kill backend mid-publish → batch stays VALIDATED, zero partial rows, re-publish succeeds
- Load: 5,000-row file → validation completes, publish chunks commit, /patients stays <2s
